## 1. Goal & end state

Prettier (formatter), markdownlint (Markdown structural linter), and EditorConfig (cross-editor floor) govern all supported structured-text in the repo, enforced by CI. The completion gate is the standard's non-mutating **check contract**:

```bash
npx prettier --check .
npx markdownlint-cli2 "**/*.md"
```

The repo mirrors the standards repo's own dogfood: Prettier owns physical formatting of every file type it supports; markdownlint owns Markdown-only structure; the two are tuned **not to fight** (13 deliberate markdownlint deviations align to or defer to Prettier's output).

## 2. Locked scope decisions

| # | Decision | Choice | Rationale |
|---|---|---|---|
| D1 | Cycle structure | Two independent cycles; **Markdown first**, Python deferred | The standards are independent (different file types, tooling, enforcement). Markdown is the clean, high-value win; Python needs heavier scoping. |
| D2 | File carve-outs | **None** — format everything | Repo markdown is ~all first-party (143 `plugins/`, 101 `docs/`, 10 root); no large vendored/foreign bucket worth excluding. Maximum consistency. |
| D3 | Toolchain pinning | **Committed root `package.json`** (`prettier` pinned as devDep) + `package-lock.json`; `node_modules/` gitignored; `markdownlint-cli2` via npx/CI action | Exact mirror of the reference repo's dogfood (`package.json` pins only Prettier; markdownlint runs via the action in CI and npx locally). |
| D4 | Prettier scope | **Faithful** — Prettier formats `md` + `json` + `yaml` repo-wide | The standard's `prettier .` cannot be scoped to Markdown without an ADR deviation (§14). Retabbing JSON/YAML is semantically inert (Zod validates manifest fields, not whitespace; GitHub Actions ignore indentation). |

## 3. Blast radius (measured)

- **254 `.md`** files — reflowed by Prettier (`proseWrap: never` puts each prose block on one line) and structurally normalized by markdownlint.
- **~30 `.json`** files — **retabbed from 2-space to tabs** (`useTabs: true`), including all 6 `plugins/*/.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json`, and both `.mcp.json`. Field names/values unchanged → no install or Zod-validation impact.
- **9 `.yaml`/`.yml`** files — retabbed (GitHub workflows, configs). Behavior-preserving.

## 5. Rollout approach

**One branch, CI sealed last, mechanical reformat isolated from hand edits.** Commit sequence within the branch:

1. **Configs + `package.json`** — add all config files; `npm install` to generate `package-lock.json`; `.gitignore += node_modules/`.
2. **Bulk reformat** (isolated commit) — `npx prettier --write .` then `npx markdownlint-cli2 --fix "**/*.md"`. This is the large mechanical diff; isolating it keeps `git show` reviewable.
3. **Any residual content fixes** (only if §6 verification surfaces them) — hand-fixed, as a separate small commit.
4. **CI + AGENTS.md + `.project-standards.yml` + `.vscode/`** — add the enforcing workflows last, confirm green.

**Why CI last:** wiring the enforcing workflows before the fix pass would make the first CI run red on 254 legacy files. Ordering keeps `main` green throughout.

*Alternatives rejected:* a single mega-commit (un-reviewable ~293-file diff: 254 md + ~30 json + 9 yaml); separate PRs per file-type (defeats the standard's "they move together" intent).

## 6. Verification (completion gate)

Run locally and in CI:

```bash
npx prettier --check .
npx markdownlint-cli2 "**/*.md"
```

Plus post-retab sanity:

- All 6 `plugin.json` + `marketplace.json` + both `.mcp.json` still parse as JSON.
- `scripts/validate-marketplace.sh` passes (marketplace schema unaffected by whitespace).
- The two existing nested `pyproject.toml`-bearing plugin test suites are untouched (Prettier does not process `.py` or `.toml`).

**Residual-violation expectation:** the rule set is tuned so the formatter and linter do not conflict, so the fix pass is expected to yield a clean tree. The check contract is the confirmation. Any stray content-rule hit (e.g. a bare URL under MD034, a missing code-fence language under MD040) is treated as a **fix-the-markdown** content bug per the standard — not a config change. An ADR exception (standard §14) is opened only if a rule proves *systemically* incompatible with house conventions; none is anticipated.

## 7. Known follow-up (flagged, out of scope this cycle)

`docs/handoff/state.md` is regenerated each session by the up-docs **propagate-repo** agent under a 2 KB cap. Once CI enforces Prettier, that agent (or the session-end ritual) must emit Prettier-clean output, or it re-dirties the file and fails CI on the next session. **Tracked as a follow-up** — it belongs with the later Python/agent cycle or a separate small fix, not this Markdown cycle.

## 8. Out of scope

- The **python-tooling** standard (its own later cycle; the `TODO.md` box stays unchecked).
- The **markdown-frontmatter** standard (a different standard; the 82 frontmatter-bearing files are not validated against its schema here).
- Fixing the `state.md` generator (§7).
- The 15 pre-existing failing pytest tests (unrelated to Markdown tooling).
- The nested plugin `package.json` (`home-assistant-dev/mcp-server`) and nested `pyproject.toml` files — Prettier may reformat their JSON/YAML but their toolchains are otherwise untouched.

## 9. Success criteria

1. `npx prettier --check .` and `npx markdownlint-cli2 "**/*.md"` both exit clean locally.
2. Both CI workflows are green on the branch before merge.
3. Marketplace + plugin manifests validate post-retab (`scripts/validate-marketplace.sh`).
4. `AGENTS.md` carries the tooling block so future agents run the fix pass.
5. `TODO.md` "Adopt markdown-tooling" checked; `docs/handoff/specs-plans.md` updated; follow-up (§7) recorded.
